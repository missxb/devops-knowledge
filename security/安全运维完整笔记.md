# 安全运维完整学习笔记

## 一、安全原则

### 1.1 纵深防御 (Defense in Depth)
```
外部 → WAF/CDN → 防火墙 → 网络隔离 → 主机安全 → 容器安全 → 应用安全 → 数据安全
```
每一层都有独立的安全措施，任何一层被突破，其他层仍能提供保护。

### 1.2 最小权限原则 (Principle of Least Privilege)
- 只授予完成工作所需的最小权限
- 定期审查和回收权限
- 使用角色而非个人授权

### 1.3 零信任模型 (Zero Trust)
- 永不信任，始终验证
- 每次访问都需要认证和授权
- 微分段网络隔离

## 二、网络安全

### 2.1 防火墙
| 类型 | 说明 | 工具 |
|------|------|------|
| 网络层 | IP/端口过滤 | iptables/nftables |
| 应用层 | HTTP 过滤 | WAF (ModSecurity/Nginx) |
| 云防火墙 | 云平台安全组 | 阿里云安全组/AWS SG |

### 2.2 网络隔离
```bash
# Kubernetes NetworkPolicy - 只允许前端访问后端
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-policy
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - port: 8080
```

### 2.3 TLS/SSL
```bash
# 生成自签证书
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout server.key -out server.crt \
  -subj "/CN=example.com"

# 检查证书信息
openssl x509 -in server.crt -text -noout

# 测试 TLS 连接
openssl s_client -connect example.com:443
```

## 三、主机安全

### 3.1 SSH 安全加固
```bash
# /etc/ssh/sshd_config
Port 2222                    # 修改默认端口
PermitRootLogin no           # 禁止root登录
PasswordAuthentication no    # 禁止密码登录，只允许密钥
MaxAuthTries 3               # 最大尝试次数
AllowUsers deploy            # 只允许特定用户
```

### 3.2 系统加固
```bash
# 1. 及时更新
yum update -y / apt upgrade -y

# 2. 配置 fail2ban 防暴力破解
yum install -y fail2ban
systemctl enable fail2ban

# 3. 审计日志
auditctl -w /etc/passwd -p wa -k passwd_changes
auditctl -w /etc/shadow -p wa -k shadow_changes

# 4. 文件完整性检查
rpm -Va  # 检查RPM包文件是否被篡改
```

### 3.3 漏洞扫描
```bash
# Trivy 镜像漏洞扫描
trivy image nginx:latest
trivy fs --security-checks config .

# Lynis 系统审计
lynis audit system
```

## 四、容器安全

### 4.1 镜像安全
```dockerfile
# ✅ 安全的 Dockerfile
FROM alpine:3.19          # 最小基础镜像
RUN apk --no-cache add ca-certificates
COPY --from=builder /app/server /usr/local/bin/server
USER nobody:nobody        # 非root运行
ENTRYPOINT ["server"]

# ❌ 不安全的 Dockerfile
FROM ubuntu:latest        # 不固定版本
RUN apt-get install ...   # 不清理缓存
USER root                 # root运行
```

### 4.2 运行时安全
```yaml
# Pod Security Standards
apiVersion: v1
kind: Pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 2000
  containers:
    - name: app
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
      resources:
        limits:
          cpu: "500m"
          memory: "256Mi"
```

### 4.3 镜像签名与验证
```bash
# Cosign 签名
cosign sign --key cosign.key harbor.example.com/myapp:v1

# 验证签名
cosign verify --key cosign.pub harbor.example.com/myapp:v1
```

## 五、Kubernetes 安全

### 5.1 RBAC 授权
```yaml
# 只允许 dev 命名空间的 deploy 用户部署应用
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: deployer
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "create", "update", "patch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: deployer-binding
  namespace: dev
subjects:
  - kind: User
    name: deploy
roleRef:
  kind: Role
  name: deployer
  apiGroup: rbac.authorization.k8s.io
```

### 5.2 Secret 管理
```bash
# K8s Secret（Base64，非加密）
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='P@ssw0rd'

# Sealed Secrets（加密存储到Git）
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# 外部密钥管理（推荐）
# HashiCorp Vault / 阿里云KMS / AWS Secrets Manager
```

### 5.3 准入控制 (Admission Control)
- **OPA/Gatekeeper**：策略引擎，强制安全策略
- **Kyverno**：K8s 原生策略引擎
- **Pod Security Standards**：K8s 内置安全标准

|------|------|----------|
| 1 | 访问控制失效 | RBAC、最小权限 |

### 6.2 密钥管理最佳实践
1. **永远不要硬编码密钥**（代码、Dockerfile、配置文件）
2. **使用密钥管理服务**（Vault、KMS）
3. **定期轮换密钥**
4. **审计密钥使用**
5. **泄露后立即吊销**

## 八、面试高频题

### Q1: 什么是纵深防御？
**答：** 在多个层次部署独立的安全措施：网络层（防火墙、WAF）、主机层（SSH加固、漏洞修补）、容器层（最小镜像、非root运行）、应用层（输入验证、认证授权）、数据层（加密、备份）。任何一层被突破，其他层仍能提供保护。

### Q2: K8s 中如何实现最小权限？
**答：** ① RBAC：为每个用户/服务账号创建最小权限的 Role；② ServiceAccount：每个应用使用独立的 ServiceAccount；③ NetworkPolicy：限制 Pod 间网络访问；④ Pod Security：禁止特权容器、只读文件系统；⑤ 资源配额：限制命名空间资源使用。

### Q3: 容器镜像安全怎么做？
**答：** ① 使用最小基础镜像（Alpine/Distroless）；② 多阶段构建，最终镜像不含构建工具；③ 镜像漏洞扫描（Trivy/Snyk）；④ 固定镜像版本，不用 latest；⑤ 镜像签名验证（Cosign）；⑥ 使用可信的镜像仓库。

## 九、学习心得
- 安全不是事后补丁，而是设计时就考虑的
- 最小权限原则说起来简单，做起来需要系统设计
- 容器安全重点在镜像安全和运行时安全
- K8s RBAC 是基础，必须掌握
- 密钥管理是最容易出问题的环节，一定要用专业工具
