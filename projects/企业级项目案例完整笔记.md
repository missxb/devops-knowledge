# 企业级项目案例完整笔记

## 项目一：微服务电商平台

### 1.1 架构设计
```
用户 → CDN → API Gateway → 微服务集群
                              ├── 用户服务
                              ├── 商品服务
                              ├── 订单服务
                              ├── 支付服务
                              ├── 库存服务
                              └── 通知服务
                                    ↓
                              数据层 (MySQL/Redis/ES/MQ)
```

### 1.2 技术栈
| 层级 | 技术 |
|------|------|
| 网关 | Kong / Nginx Ingress |
| 服务框架 | Spring Cloud / Go Kit |
| 服务注册 | Nacos / Consul |
| 配置中心 | Nacos / Apollo |
| 链路追踪 | Jaeger / SkyWalking |
| 消息队列 | RocketMQ / Kafka |
| 数据库 | MySQL (分库分表) + Redis |
| 搜索 | Elasticsearch |
| 容器编排 | Kubernetes |
| CI/CD | GitLab CI + ArgoCD |
| 监控 | Prometheus + Grafana + ELK |

### 1.3 关键挑战与解决方案
| 挑战 | 方案 |
|------|------|
| 分布式事务 | Seata / Saga 模式 |
| 服务雪崩 | 熔断(Hystrix) + 限流(Sentinel) + 降级 |
| 数据一致性 | 最终一致性 + 补偿机制 |
| 高并发 | 缓存(Redis) + 异步(MQ) + 读写分离 |
| 服务治理 | 服务网格(Istio/ASM) |

### 1.4 部署架构
```
K8s 集群 (3 Master + N Worker)
├── Namespace: production
│   ├── Deployment: user-service (3 replicas)
│   ├── Deployment: order-service (5 replicas)
│   ├── StatefulSet: mysql-master-slave
│   └── DaemonSet: log-agent
├── Namespace: staging
│   └── ... (类似结构，资源减半)
└── Namespace: monitoring
    ├── Prometheus
    ├── Grafana
    └── ELK
```

### 1.5 性能指标
| 指标 | 目标 |
|------|------|
| QPS | > 10,000 |
| P99延迟 | < 200ms |
| 可用性 | 99.95% |
| 数据丢失 | 0 |

## 项目二：SaaS 多租户平台

### 2.1 多租户架构模式
| 模式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 独立数据库 | 每租户一个DB | 隔离性最强 | 成本高 |
| 共享数据库独立Schema | 每租户一个Schema | 中等隔离 | 迁移复杂 |
| 共享Schema | 所有租户共用表 | 成本最低 | 隔离性差 |

### 2.2 资源隔离
```yaml
# K8s ResourceQuota - 限制租户资源
apiVersion: v1
kind: ResourceQuota
metadata:
  name: tenant-a-quota
  namespace: tenant-a
spec:
  hard:
    requests.cpu: "10"
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "50"
    services: "20"
```

### 2.3 计费设计
```
按量计费：API调用次数 + 存储用量 + 计算资源
包年包月：固定套餐（基础版/专业版/企业版）
```

## 项目五：混合云灾备

### 5.1 灾备架构
```
主站（阿里云华东）           灾备站（阿里云华南）
┌──────────────┐           ┌──────────────┐
│ K8s 集群      │    DTS    │ K8s 集群      │
│ MySQL 主库    │ ───────→ │ MySQL 从库    │
│ Redis 集群    │    同步    │ Redis 集群    │
│ 应用服务      │           │ 应用服务(待命) │
└──────────────┘           └──────────────┘
       ↑                          ↑
       └──────── GSLB ────────────┘
           (全局流量管理)
```

### 5.2 RPO/RTO 目标
| 指标 | 含义 | 目标值 |
|------|------|--------|
| RPO | 可容忍的数据丢失时间 | < 5秒 |
| RTO | 恢复服务所需时间 | < 5分钟 |

### 5.3 演练计划
- **每季度**：模拟主站故障，切换流量到灾备站
- **每年**：全链路灾备演练，验证数据一致性
- **复盘**：每次演练后复盘，优化流程

## 面试高频题

### Q1: 如何设计一个高可用架构？
**答：** 从四个层面考虑：① 接入层：多可用区部署 + 全局负载均衡；② 应用层：无状态服务 + 多副本 + 自动伸缩；③ 数据层：主从复制 + 读写分离 + 跨AZ容灾；④ 运维层：完善的监控告警 + 自动故障转移。目标是任何单点故障都不影响服务。

### Q2: 微服务拆分的原则是什么？
**答：** ① 单一职责：每个服务只负责一个业务领域；② 高内聚低耦合：服务内部紧密相关，服务之间松散依赖；③ 按业务能力拆分：用户、订单、支付各自独立；④ 数据独立：每个服务有自己的数据库；⑤ 渐进式拆分：先拆大模块，再逐步细化。

### Q3: 如何保证微服务间的数据一致性？
**答：** 不追求强一致性，采用最终一致性。方案：① Saga 模式：每个服务执行本地事务，失败时执行补偿操作；② 事务消息：RocketMQ 事务消息确保本地事务和消息发送的原子性；③ 事件溯源：记录所有事件，需要时重放。关键是设计好补偿机制和幂等接口。

## 学习心得
- 项目经验比理论知识更重要，面试官看重的是解决实际问题的能力
- 架构设计没有银弹，要根据业务规模和团队能力选择合适的方案
- 高可用、高性能、可扩展是架构设计的三个永恒主题
- 灾备和容灾容易被忽视，但出了问题就是大问题
- DevOps 平台建设是提升团队效率的关键投资
