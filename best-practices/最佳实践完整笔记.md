# DevOps 最佳实践完整笔记

## 一、文化实践

### 1.1 建立协作文化
- **跨职能团队**：开发+测试+运维在一个团队，共同负责交付
- **共同目标**：团队 KPI 统一（如交付频率、线上故障率），不是各自的指标
- **无指责事后分析 (Blameless Postmortem)**：出问题后关注"怎么修"而不是"谁的错"
- **知识共享**：定期技术分享、结对编程、轮岗制度

### 1.2 沟通机制
| 机制 | 频率 | 目的 |
|------|------|------|
| 每日站会 | 每天 | 同步进度、暴露阻碍 |
| 技术分享 | 每周 | 知识传播、团队成长 |
| 复盘会议 | 每次发布后 | 总结经验、改进流程 |
| 1-on-1 | 每月 | 个人发展、团队健康 |

## 二、代码管理最佳实践

### 2.1 分支策略
```
main (生产)
  ↑
develop (开发主线)
  ↑
feature/xxx (功能分支)
```

**Git Flow vs Trunk-Based：**
| 策略 | 特点 | 适用场景 |
|------|------|----------|
| Git Flow | 多分支（feature/release/hotfix） | 发布周期长的传统项目 |
| Trunk-Based | 主干开发，短命分支 | 持续交付的互联网项目 |

### 2.2 代码审查 (Code Review)
**原则：**
- 每个 PR 至少一人审查
- 关注逻辑正确性、安全性、性能
- 不要审查超过 400 行的 PR（拆分）
- 使用自动化工具（lint、格式化）减少人工审查负担

### 2.3 提交规范
```
feat: 新功能
fix: 修复bug
docs: 文档变更
style: 代码格式（不影响功能）
refactor: 重构
test: 测试相关
chore: 构建/工具变更
```

## 三、测试最佳实践

### 3.1 测试金字塔
```
         /  E2E测试  \         少量，验证关键流程
        / 集成测试     \        适量，验证模块交互
       / 单元测试       \       大量，验证单个函数
```

### 3.2 测试覆盖率目标
| 类型 | 覆盖率目标 | 执行频率 |
|------|-----------|---------|
| 单元测试 | > 80% | 每次提交 |
| 集成测试 | 核心流程 100% | 每次合并 |
| E2E测试 | 关键路径 100% | 每日/每次发布 |
| 性能测试 | 关键接口 | 每周/每次发布 |

### 3.3 测试自动化
```python
# pytest 示例
import pytest

class TestUserService:
    def test_create_user(self):
        user = create_user("test@example.com", "password123")
        assert user.email == "test@example.com"
        assert user.is_active is True

    def test_duplicate_email_raises(self):
        create_user("test@example.com", "pass1")
        with pytest.raises(DuplicateEmailError):
            create_user("test@example.com", "pass2")

    @pytest.mark.parametrize("email,expected", [
        ("valid@example.com", True),
        ("invalid", False),
        ("", False),
    ])
    def test_email_validation(self, email, expected):
        assert validate_email(email) == expected
```

## 四、部署最佳实践

### 4.1 部署策略对比
| 策略 | 原理 | 回滚速度 | 资源开销 | 风险 |
|------|------|----------|----------|------|
| 滚动更新 | 逐步替换Pod | 慢 | 低 | 中 |
| 蓝绿部署 | 两套环境切换 | 秒级 | 高（2倍） | 低 |
| 金丝雀 | 小比例流量验证 | 快 | 低 | 低 |
| A/B测试 | 按用户特征分流 | 快 | 中 | 低 |

### 4.2 金丝雀部署示例 (Istio)
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts: [myapp]
  http:
    - route:
        - destination:
            host: myapp
            subset: stable
          weight: 90
        - destination:
            host: myapp
            subset: canary
          weight: 10
```

### 4.3 版本管理
- **语义化版本**：`MAJOR.MINOR.PATCH`（如 2.1.3）
- **镜像标签**：使用 Git SHA 或构建号，不要用 `latest`
- **配置版本化**：配置和代码一起版本控制

## 五、运维最佳实践

### 5.1 可观测性三支柱
| 支柱 | 工具 | 用途 |
|------|------|------|
| **指标 (Metrics)** | Prometheus + Grafana | 趋势分析、告警 |
| **日志 (Logging)** | ELK / Loki | 问题排查、审计 |
| **链路 (Tracing)** | Jaeger / Tempo | 调用链分析、性能瓶颈 |

### 5.2 事故管理流程
```
检测 → 响应 → 恢复 → 复盘 → 改进
  ↓       ↓       ↓       ↓       ↓
告警通知  升级处理  服务恢复  事后分析  跟踪改进
```

### 5.3 容量规划
- **基准测试**：了解系统当前容量上限
- **增长预测**：根据业务增长预测资源需求
- **弹性策略**：设置自动伸缩规则
- **预算管理**：提前申请资源预算

## 六、安全最佳实践 (DevSecOps)

### 6.1 安全左移
```
传统：开发 → 测试 → 部署 → 安全检查（太晚了！）
DevSecOps：开发(安全编码) → 测试(自动扫描) → 部署(策略检查) → 运行时(监控)
```

### 6.2 安全自动化
| 阶段 | 工具 | 检查内容 |
|------|------|---------|
| 编码 | SonarLint | 代码质量、安全漏洞 |
| 提交 | pre-commit hooks | 密钥泄露检查 |
| 构建 | Trivy/Snyk | 依赖漏洞 |
| 部署 | OPA/Kyverno | 安全策略 |
| 运行时 | Falco | 异常行为检测 |

## 七、面试高频题

### Q1: 如何推动 DevOps 转型？
**答：** 分四步：① 获得管理层支持，明确目标；② 选一个试点项目，先跑通 CI/CD；③ 建立自动化测试和监控；④ 逐步推广到其他项目，建立度量体系。关键是"小步快跑"，不要一步到位。常见错误：只买工具不改文化。

### Q2: 什么是"左移"？在实践中怎么做？
**答：** 把质量保障活动尽早放到开发前端。实践：① IDE 插件实时检查代码质量；② PR 触发自动化测试和安全扫描；③ 预发布环境自动验证；④ 配置即代码，变更可审查。越早发现问题，修复成本越低（生产环境修复成本是编码阶段的100倍）。

### Q3: 如何平衡发布速度和系统稳定性？
**答：** 用 SRE 的错误预算机制。设定 SLO（如 99.9% 可用性），计算错误预算（每月约43分钟）。预算充足时大胆发布新功能，预算不足时冻结变更专注稳定性。这样团队有共同的量化标准，避免"开发要快"和"运维要稳"的矛盾。

## 八、学习心得
- DevOps 的核心是文化，工具只是手段
- CI/CD 是基础，没有自动化就没有 DevOps
- 测试金字塔指导测试投入比例，单元测试是基石
- 安全左移可以大幅降低安全风险和修复成本
- 度量驱动改进，DORA 指标是衡量 DevOps 效果的金标准
