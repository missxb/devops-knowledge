# 容器技术完整学习笔记

## 一、Docker 核心知识

### 1.1 容器 vs 虚拟机
| 特性 | 容器 | 虚拟机 |
|------|------|--------|
| 启动速度 | 秒级 | 分钟级 |
| 资源占用 | MB级 | GB级 |
| 隔离性 | 进程级（Namespace+Cgroup） | 硬件级（Hypervisor） |
| 性能 | 接近原生 | 有损耗 |
| 密度 | 单机可运行数百个 | 单机十几个 |
| 镜像大小 | 通常几十MB | 通常几GB |

### 1.2 Docker 核心架构
```
Docker Client (docker命令)
       ↓
Docker Daemon (dockerd)
       ↓
  ┌────┴────┐
Containerd    镜像存储
  ↓          (Image Registry)
Runc
  ↓
Linux Kernel (Namespace + Cgroups)
```

### 1.3 核心概念
- **镜像 (Image)**：只读模板，分层存储（UnionFS）
- **容器 (Container)**：镜像的运行实例，可写层 + 只读层
- **仓库 (Registry)**：存储镜像，如 Docker Hub、Harbor
- **Dockerfile**：构建镜像的脚本

### 1.4 Dockerfile 最佳实践
```dockerfile
# ✅ 多阶段构建 - 减小镜像体积
FROM golang:1.21 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server .

FROM alpine:3.19
RUN apk --no-cache add ca-certificates tzdata
COPY --from=builder /app/server /usr/local/bin/server
EXPOSE 8080
USER nobody:nobody
ENTRYPOINT ["server"]
```

**关键原则：**
1. 使用多阶段构建，最终镜像只包含运行时依赖
2. 合并 RUN 指令减少层数
3. 利用缓存：不常变的指令放前面
4. 使用 .dockerignore 排除不需要的文件
5. 非 root 用户运行
6. 固定基础镜像版本（不用 latest）

### 1.5 Docker 网络模式
| 模式 | 说明 | 适用场景 |
|------|------|----------|
| bridge | 默认，Docker创建虚拟网桥 | 单机多容器通信 |
| host | 共享宿主机网络栈 | 高性能网络需求 |
| none | 无网络 | 安全隔离场景 |
| overlay | 跨主机容器通信 | Docker Swarm 集群 |
| macvlan | 容器有独立MAC地址 | 需要独立IP的场景 |

### 1.6 Docker 存储
- **Volume**：Docker管理的存储，持久化数据首选
- **Bind Mount**：挂载宿主机目录
- **tmpfs**：内存临时存储

## 二、Kubernetes 核心知识

### 2.1 架构总览
```
┌──────────────── Master Node ────────────────────┐
│  API Server ← Scheduler ← Controller Manager    │
│       ↕                                          │
│     etcd                                        │
└─────────────────────────────────────────────────┘
          ↓
┌──────── Worker Node 1 ────────┐ ┌── Worker Node 2 ──┐
│  Kubelet  Kube-Proxy  Runtime  │ │   ...              │
│  ┌─────┐ ┌─────┐ ┌─────┐     │ │                    │
│  │Pod A│ │Pod B│ │Pod C│     │ │                    │
│  └─────┘ └─────┘ └─────┘     │ │                    │
└───────────────────────────────┘ └────────────────────┘
```

### 2.2 核心资源对象
| 资源 | 作用 | 关键点 |
|------|------|--------|
| **Pod** | 最小调度单位 | 一个或多个容器，共享网络和存储 |
| **Deployment** | 无状态应用部署 | 管理ReplicaSet，支持滚动更新 |
| **StatefulSet** | 有状态应用部署 | 稳定的网络标识和持久存储 |
| **DaemonSet** | 每节点运行一个Pod | 日志收集、监控Agent |
| **Service** | 服务发现和负载均衡 | ClusterIP/NodePort/LoadBalancer |
| **Ingress** | HTTP路由入口 | 域名和路径路由 |
| **ConfigMap** | 配置数据 | 键值对或文件 |
| **Secret** | 敏感数据 | Base64编码（非加密） |
| **PV/PVC** | 持久存储 | PV是存储资源，PVC是存储请求 |
| **HPA** | 水平自动伸缩 | 基于CPU/内存/自定义指标 |

### 2.3 Pod 生命周期
```
Pending → Running → Succeeded/Failed
  ↓
  ├── 调度中（等待资源）
  └── 拉取镜像中

Init Containers → Main Containers → PostStart Hook
                                     ↓
                                  Readiness Probe → 加入Service
                                  Liveness Probe → 重启检测
```

### 2.4 Service 类型
| 类型 | 说明 | 使用场景 |
|------|------|----------|
| ClusterIP | 集群内部IP | 内部服务通信（默认） |
| NodePort | 节点端口暴露 | 开发测试 |
| LoadBalancer | 云LB | 生产环境对外服务 |
| ExternalName | CNAME映射 | 访问外部服务 |

### 2.5 资源管理
```yaml
resources:
  requests:        # 调度依据，保证分配
    cpu: "250m"    # 0.25核
    memory: "256Mi"
  limits:          # 运行上限，超出被限制/OOMKilled
    cpu: "500m"
    memory: "512Mi"
```

### 2.6 滚动更新策略
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%        # 最多多出25%的Pod
    maxUnavailable: 25%  # 最多25%的Pod不可用
```

## 三、Helm 包管理

### 3.1 核心概念
- **Chart**：应用包，包含模板和默认配置
- **Release**：Chart 的运行实例
- **Repository**：Chart 的存储仓库
- **Values**：覆盖默认配置的参数

### 3.2 常用命令
```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo nginx
helm install my-nginx bitnami/nginx -n production
helm upgrade my-nginx bitnami/nginx -f values.yaml
helm rollback my-nginx 1
helm list -A
```

## 四、面试高频题

### Q1: Docker 和虚拟机的区别？
**答：** Docker 容器共享宿主机内核，通过 Linux Namespace 隔离进程视图，Cgroups 限制资源，启动秒级、体积小。虚拟机有独立内核，通过 Hypervisor 虚拟硬件，启动分钟级、隔离性更强但资源开销大。容器适合微服务，虚拟机适合强隔离需求。

### Q2: Pod 和容器的关系？
**答：** Pod 是 K8s 最小调度单位，可以包含一个或多个容器。同一个 Pod 内的容器共享网络（同一个 IP）和存储（可共享 Volume），通常用 sidecar 模式组合（如主容器+日志收集容器）。

### Q3: Deployment 和 StatefulSet 的区别？
**答：** Deployment 适合无状态应用，Pod 名称随机、可任意替换。StatefulSet 适合有状态应用（如数据库），Pod 有稳定名称（pod-0, pod-1）、有序部署/销毁、每个 Pod 有独立的 PVC。

### Q4: Service 是如何实现负载均衡的？
**答：** K8s Service 通过 kube-proxy 组件实现。kube-proxy 监听 Service 和 Endpoints 变化，在每个节点上维护 iptables/IPVS 规则，将访问 Service ClusterIP 的流量随机（或轮询）转发到后端的某个 Pod。

### Q5: ConfigMap 和 Secret 的区别？
**答：** 功能类似，都是注入配置。区别是 Secret 的值是 Base64 编码的（注意不是加密），用于敏感数据如密码、Token。ConfigMap 用于非敏感配置。两者都可以以环境变量或文件挂载方式注入容器。

## 五、学习心得
- Docker 是基础，Dockerfile 优化是必备技能
- K8s 资源对象很多，先掌握 Pod/Deployment/Service/Ingress 四件套
- Helm 是 K8s 的"包管理器"，生产环境强烈推荐
- 容器安全很重要：非 root 运行、镜像扫描、网络策略
- 实践出真知，minikube/kind 搭建本地集群多练
