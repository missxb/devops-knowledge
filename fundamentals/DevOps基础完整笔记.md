# DevOps 基础知识 - 完整学习笔记

## 一、DevOps 定义与起源

### 1.1 什么是 DevOps
DevOps = Development（开发）+ Operations（运维），是一种**文化和实践**，强调开发和运维团队之间的协作、自动化和持续交付，目标是缩短软件开发生命周期，提高交付频率和质量。

### 1.2 为什么需要 DevOps
**传统模式的问题：**
- 开发追求变更（快速上线新功能）
- 运维追求稳定（减少变更降低风险）
- 两个目标天然矛盾，导致互相推诿

**DevOps 解决的问题：**
- 打破开发和运维之间的"墙"
- 自动化重复性工作
- 建立快速反馈机制
- 实现持续交付和持续改进

### 1.3 DevOps 与敏捷的关系
| 概念 | 关注点 | 范围 |
|------|--------|------|
| 敏捷 (Agile) | 开发阶段的迭代和协作 | 开发团队 |
| DevOps | 从开发到运维的全链路自动化 | 开发+运维+测试 |
| SRE | 用软件工程方法解决运维问题 | 运维团队 |

三者互补：敏捷管开发流程，DevOps 管交付管道，SRE 管运行时可靠性。

## 二、核心原则 - CALMS 模型

### 2.1 Culture（文化）
- **信任与协作**：开发和运维共同对交付结果负责
- **无指责文化 (Blameless)**：出问题时关注"怎么修"而不是"谁的错"
- **共享目标**：团队有共同的 KPI（如交付频率、故障恢复时间）

### 2.2 Automation（自动化）
- **构建自动化**：代码提交自动触发构建
- **测试自动化**：单元测试、集成测试、E2E 测试
- **部署自动化**：一键部署或全自动部署
- **基础设施自动化**：IaC（Infrastructure as Code）

### 2.3 Lean（精益）
- **消除浪费**：减少手动审批、重复操作
- **小批量交付**：频繁小变更优于大批量发布
- **价值流映射**：找出交付流程中的瓶颈

### 2.4 Measurement（度量）
**四个关键指标 (DORA Metrics)：**
| 指标 | 精英团队 | 低效团队 |
|------|----------|----------|
| 部署频率 | 每天多次 | 每月或更低 |
| 变更前置时间 | < 1小时 | 1-6个月 |
| 变更失败率 | < 5% | 31-45% |
| 故障恢复时间 | < 1小时 | 1周-1月 |

### 2.5 Sharing（分享）
- 知识共享：文档、Wiki、技术分享会
- 工具共享：统一的工具链和平台
- 经验分享：复盘会议、事后分析

## 四、DevOps 成熟度模型

### 4.1 五个级别
```
Level 1: 初始级 - 手动操作，无自动化
Level 2: 基础级 - 有CI，基本自动化构建
Level 3: 标准级 - CI/CD完整，自动化测试
Level 4: 管理级 - 监控完善，度量驱动改进
Level 5: 优化级 - 持续改进，全自动交付
```

### 4.2 自评清单
- [ ] 代码版本控制覆盖率 100%
- [ ] CI 流水线覆盖率 > 90%
- [ ] 自动化测试覆盖率 > 80%
- [ ] 部署自动化率 > 90%
- [ ] 平均故障恢复时间 < 1小时
- [ ] 部署频率 > 每天1次

## 五、SRE（Site Reliability Engineering）

### 5.1 SRE vs DevOps
| 维度 | DevOps | SRE |
|------|--------|-----|
| 出发点 | 文化和实践 | 工程方法论 |
| 关注点 | 交付流程 | 可靠性 |
| 衡量标准 | DORA指标 | SLI/SLO/SLA |
| 来源 | 社区实践 | Google |

### 5.2 SLI/SLO/SLA
- **SLI (Service Level Indicator)**：服务级别指标，如请求延迟、错误率
- **SLO (Service Level Objective)**：服务级别目标，如 99.9% 可用性
- **SLA (Service Level Agreement)**：服务级别协议，包含违约赔偿

### 5.3 错误预算 (Error Budget)
```
错误预算 = 1 - SLO
例如 SLO = 99.9%，则错误预算 = 0.1% = 每月约43分钟不可用

错误预算充足 → 可以大胆发布新功能
错误预算不足 → 冻结变更，专注稳定性
```

## 六、面试高频题

### Q1: 什么是 DevOps？和敏捷有什么区别？
**答：** DevOps 是一种文化和实践，强调开发和运维的协作与自动化，覆盖从代码提交到生产运行的全生命周期。敏捷关注开发阶段的迭代交付，DevOps 把敏捷的理念延伸到了运维。简单说：敏捷管开发，DevOps 管交付。

### Q2: 如何衡量 DevOps 的效果？
**答：** 使用 DORA 四个关键指标：① 部署频率（多久部署一次）；② 变更前置时间（从提交到上线的时间）；③ 变更失败率（多少部署导致故障）；④ 故障恢复时间（从故障到恢复的时间）。精英团队能做到每天多次部署、变更失败率 < 5%。

### Q3: DevOps 如何在团队中落地？
**答：** 分阶段推进：① 先建立版本控制和 CI（代码自动构建和测试）；② 再建立 CD（自动化部署）；③ 然后完善监控和告警；④ 最后建立度量体系持续改进。关键是文化先行，工具跟上，不要一步到位。

### Q4: SRE 和 DevOps 有什么关系？
**答：** 两者目标一致（提高交付效率和系统可靠性），但方法不同。DevOps 是一套文化和实践，没有固定实现方式；SRE 是 Google 提出的工程方法论，有明确的 SLI/SLO/SLA、错误预算等概念。可以把 SRE 看作 DevOps 的一种具体实现。

### Q5: 什么是"左移"（Shift Left）？
**答：** 把质量保障活动尽早放到开发流程的前端。比如：在编码阶段就做静态分析，而不是等到测试阶段才发现问题；在设计阶段就考虑安全，而不是上线前才做安全审计。越早发现问题，修复成本越低。

## 七、学习心得
- DevOps 不是某个工具，而是一种文化转变
- 自动化是 DevOps 的核心手段，但文化是基础
- DORA 指标是衡量 DevOps 效果的金标准
- SRE 用工程方法解决了运维的"凭感觉"问题
- DevOps 转型需要循序渐进，不能一步到位
